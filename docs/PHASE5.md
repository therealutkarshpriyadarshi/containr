### Maintenance

#### Security Patch Process

See [SECURITY.md](../SECURITY.md) for full details.

**Process:**
1. **Private Disclosure**
   - Report received via security advisory
   - Initial assessment within 48 hours
   - Severity classification

2. **Fix Development**
   - Private repository for fix
   - Thorough testing
   - Security review

3. **Coordinated Release**
   - Agree on disclosure date with reporter
   - Prepare security advisory
   - Release fixed version
   - Public announcement

4. **Post-Release**
   - Monitor for issues
   - Backport to supported versions
   - Update security documentation

#### Dependency Updates

**Automated Updates:**
- Dependabot enabled
- Weekly dependency checks
- Automated PR creation
- Security vulnerability scanning

**Manual Review:**
- Review automated PRs
- Test compatibility
- Update documentation
- Merge and release

**Tools:**
- `go get -u ./...` - Update all dependencies
- `go mod tidy` - Clean up
- `govulncheck` - Vulnerability scanning
- `nancy` - Dependency security audit

#### Long-Term Support

**Support Policy:**
- **Current Major Version:** Full support
- **Previous Major Version:** Security fixes (6 months)
- **Older Versions:** Unsupported

**Release Schedule:**
- **Major:** Annually (or when ready)
- **Minor:** Quarterly (3-4 months)
- **Patch:** As needed (security: immediate)

**Deprecation Policy:**
1. Announce deprecation with migration path
2. Mark as deprecated in code and docs
3. Keep deprecated feature for 2 minor versions
4. Remove in next major version
5. Update migration guide

**Example Timeline:**
- v1.0.0: Feature marked deprecated
- v1.1.0: Still available with warnings
- v1.2.0: Still available with warnings
- v2.0.0: Feature removed
